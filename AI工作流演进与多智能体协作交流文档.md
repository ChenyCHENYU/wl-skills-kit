# AI 工作流演进与多智能体协作 · 交流文档

> **撰写时间**：2026-05-12  
> **定位**：对齐团队 AI 工作流演进方向，厘清当前阶段目标、多智能体分工设计、可视化工作台规划，以及前后端完整工作流最佳实践。  
> **背景项目**：wl-skills-kit（当前基线 v2.11.11，L1-L5 + MCP/Agent Pipeline 已落地，L6 Multi-Agent 进入规划期）

---

## 一、我们在哪里

当前体系已构建 5 层能力：

```
L1  提示词工程      ✅  copilot-instructions + 多编辑器适配 + standards 懒加载
L2  Skills          ✅  10 个 Skill 覆盖原型扫描 → 契约 → 生成 → 审计 → 同步 → 修复
L3  MCP Tools       ✅  17 个 Tool，菜单 / 字典 / 权限 / 项目感知 / 页面校验 / 通知
L4  CLI             ✅  9 条命令，init / update / validate / doctor-ui / export 等
L5  Agent Pipeline  🟡  协议与运行手册已落地，进入试运行阶段
L6  Multi-Agent     🔭  L5 稳定后启动，目标：无人干预自动跑完 + 可视化工作台
L7  自演化体系      🔭  需积累足够审计报告与模板样本后规划
```

## 二、核心演进策略

### 2.1 整体思路

> **开发侧先行，产品侧镜像，双向对齐，测试侧收口。**

开发侧骨架流程已清晰，不等产品侧。产品侧在条件成熟时，参照开发侧的最佳实践独立规划自己的 Agent 工作流，两侧落地后再做双向调试——这样两侧都不互相阻塞，也不会因为接口未定义而互相绑死。

```
开发侧 Pipeline（先行）
       ↕  双向调试对齐
产品侧 Pipeline（独立规划，镜像思路）
       ↕
测试侧 Pipeline（收口，依赖前两侧产物）
```

### 2.2 Multi-Agent 协作的三个前提

在进入 L6 之前，以下三点必须在 L5 阶段验证清楚：

| 前提 | 具体要求 |
|---|---|
| **Skill I/O 有明确格式** | `_pipeline.md` 里每个节点的输入/输出结构无歧义，Agent 之间传递的数据不需要靠"理解"来补全 |
| **人工确认边界有定义** | 哪些步骤 Agent 可自动执行，哪些必须展示 diff + 等待人工确认后才能继续 |
| **异常路径有回退机制** | 任意节点出错时，能干净中止，不留半成品，可从上一个稳定节点重跑 |

### 2.3 关于"节点决策记录"

L5 人工执行 Pipeline 时，开发者会自然地做一些"无意识判断"，例如：

- `prototype-scan` 输出了一个叫 `AddModal` 的页面，你一眼看出这是弹窗不是独立页面，手动调整后再传给 `page-codegen`
- `convention-audit` 报了一条警告，你知道这个项目的特殊约定所以跳过了
- 某步骤输出格式和下游期望的有点出入，你手动整理了一下

**这些"顺手处理"的判断，就是 L6 Agent 需要被显式告知的规则。**

实践方法：在 L5 跑通阶段，开一个 `_pipeline-notes.md`，每次手动调整就记一条：`"prototype-scan 输出组件名含 Modal/Dialog → 标记为 dialog-type，不走 page-codegen 主流程"`。这些笔记会直接变成 Orchestrator 的路由规则和 Agent 的判断依据。

## 三、分阶段落地路径

```
阶段一 ──── 开发侧 Pipeline 验证（当前）
阶段二 ──── 开发侧 Multi-Agent + 可视化工作台
阶段三 ──── 产品侧 + 测试侧接入，全链路闭环
```

### 阶段一：开发侧 Pipeline 验证（L5 试运行）

**原则**：先跑"只读/生成"路径，再引入"写后端"高风险路径。

```text
✅ 低风险路径（优先验证）
prototype-scan → business-doc-extract → api-contract → page-codegen
→ validate-page → convention-audit → 输出审计报告

⚠️ 高风险路径（低风险稳定后再验证）
menu-sync / dict-sync / permission-sync（写后端，需有 diff 预览和人工确认）
```

**完成标志**：3 个以上完整业务页面从需求到上线走通全链路，无需 AI 二次补救。

---

### 阶段二：开发侧 Multi-Agent + 可视化工作台（L6）

**目标**：无人干预下自动跑完完整 Pipeline，同时有可视化工作台供团队观察和介入。

#### 2.1 角色 Agent 分工

```
┌──────────────────────────────────────────────────────────────┐
│                     Orchestrator（调度层）                      │
│  • 接收需求，路由到对应 Agent 组合                                │
│  • 管理节点执行顺序，处理 Agent 间数据传递                         │
│  • 识别人工确认节点，暂停并推送通知                                │
│  • 汇总执行摘要和产物清单                                        │
└───────────┬──────────────────────────────────────────────────┘
            │
  ┌─────────▼──────────┐      ┌────────────────────┐
  │  需求解析 Agent      │      │   代码生成 Agent     │
  │  prototype-scan    │─────►│   page-codegen      │
  │  business-doc      │      │   api-contract      │
  │  → 输出：页面清单    │      │   → 输出：页面骨架    │
  │    + 业务理解文档    │      │     + api.md        │
  └────────────────────┘      └──────────┬─────────┘
                                         │
  ┌─────────────────────┐     ┌──────────▼─────────┐
  │   规范审计 Agent      │◄────│   校验 Agent        │
  │   convention-audit  │     │   validate-page     │
  │   code-fix          │     │   doctor-ui         │
  │   → 输出：审计报告    │     │   route-check       │
  └─────────────────────┘     └────────────────────┘
            │
  ┌─────────▼──────────┐
  │   同步 Agent        │
  │   menu-sync         │  ← 此节点必须有人工确认后才执行写操作
  │   dict-sync         │
  │   permission-sync   │
  │   → 输出：后台同步报告│
  └────────────────────┘
```

**每个 Agent 的定义要回答（岗位说明书思维）**：

| 要素 | 说明 |
|---|---|
| 我负责什么 | 职责范围，输入来源，产物落点 |
| 我不负责什么 | 明确不越界事项 |
| 我按什么标准交付 | 产物格式 + 质量红线 |
| 什么情况下我必须停下来 | 人工确认触发条件 |

#### 2.2 可视化工作台设计

> 这是 L6 阶段的重要配套能力，不是可选项。

**核心价值**：让工作流从"AI 在黑盒里跑"变成"团队可观测、可介入、可追溯"。

**节点状态可视化**：

```
[需求解析] ──► [代码生成] ──► [校验] ──► [规范审计] ──► [同步]
   ✅运行完      ⏳执行中       ⬜等待       ⬜等待        🔒等待确认
```

**功能规划**：

| 功能 | 说明 |
|---|---|
| **流程拓扑图** | 以节点图展示完整 Pipeline，每个节点显示状态（等待/执行中/成功/失败/等待确认） |
| **节点详情面板** | 点击任意节点，查看该节点的输入参数、输出产物、耗时和日志 |
| **人工确认入口** | 高风险节点（写后端/写文件）在此弹出 diff 预览，人工点击"确认执行"或"跳过" |
| **重跑能力** | 任意节点失败后，可从该节点重新触发，不需要从头跑 |
| **产物快捷访问** | 每个节点产物（api.md / 审计报告 / 页面文件）可直接在工作台打开 |
| **执行历史** | 记录每次 Pipeline 执行的完整快照，支持回溯和对比 |

**技术选型参考**（按成本从低到高）：

| 方案 | 适用场景 |
|---|---|
| 基于现有 MCP + Copilot Chat，配合飞书通知 | 近期过渡方案，成本最低 |
| 集成 Dify / n8n 等可视化工作流平台 | 中期，快速获得可视化能力，Agent 逻辑托管到平台 |
| 自研工作台（基于团队内部系统） | 远期，与业务系统深度集成，支持权限和审计 |

### 阶段三：产品侧 + 测试侧接入（全链路闭环）

产品侧和测试侧各自独立规划，参照开发侧 Agent 建设思路，落地后再双向对齐。

```
[产品 Agent]                    [开发侧 Multi-Agent]              [测试 Agent]
需求挖掘                                                          接口测试生成
场景还原          ──移交：page-spec + input-spec──►               E2E 场景生成
链路闭环检查                                                      回归测试补全
输出验收口径       ◄──反馈：审计报告 + 缺陷预警──                   覆盖率报告
```

**整体闭环**：

```
产品 Agent ──► 开发侧 Pipeline ──► 测试 Agent
    ▲                                  │
    └──────── 缺陷 & 需求反馈 ──────────┘
```

## 四、前端工作流（完整最佳实践）

> 基于 wl-skills-kit 当前 Skill 体系，覆盖从需求输入到上线的完整前端开发链路。

### 4.1 标准链路

```text
① 原型 / 详设 / 口述需求
         ↓
② prototype-scan
   输出：pages-list（页面清单 + 字段 + 交互说明）

         ↓
③ business-doc-extract（资料达模块级完整度时触发）
   输出：docs/business/ 业务理解文档（需求 / 字典 / 字段）

         ↓
④ api-contract
   输出：src/views/**/api.md（前后端接口契约）

         ↓
⑤ page-codegen
   输出：index.vue / data.ts / index.scss（符合 14 条规范的页面骨架）

         ↓
⑥ validate-page + doctor-ui
   检查：AGGrid / cid / api.md 存在性 / wk-skills-ui 接入状态

         ↓
⑦ convention-audit
   输出：结构化审计报告（14 条规范扫描）

         ↓
⑧ code-fix（可选，仅处理报告中明确可自动修复的项）
   → convention-audit 复扫确认

         ↓
⑨ 菜单 / 字典 / 权限闭环
   menu-sync → dict-sync → permission-sync（人工确认后写后端）

         ↓
⑩ 输出：可运行页面 + 规范报告 + 后台数据同步完毕
```

### 4.2 场景分支

| 场景 | 走哪段链路 |
|---|---|
| 有完整原型/详设 | 全链路 ① → ⑩ |
| 口述需求快速出页面 | ⑤ → ⑥ → ⑦（先跑起来，后续补契约） |
| 存量项目体检 | `wls_code_scan` → ⑦ → ⑧ → ⑥ |
| 只做菜单/权限同步 | 直接 ⑨，已有 api.md 和页面骨架为前提 |

### 4.3 质量红线

- 所有页面必须有 `api.md`，无 api.md 不进菜单同步
- AGGrid 必须用 `defineColumns()` + `renderOps()`，无 cid 不通过 validate-page
- 写后端操作（menu/dict/permission）必须有人工确认，不允许无感自动写入

## 五、后端工作流（完整最佳实践）

> 与前端工作流并行建设，覆盖从接口契约到服务上线的完整后端开发链路。  
> 当前为规划阶段，可参照前端体系逐步落地。

### 5.1 标准链路

```text
① 前端 api.md（接口契约）或产品侧输出的 input-spec
         ↓
② api-design-be（接口设计审查）
   • 检查 RESTful 命名规范
   • 确认请求/响应结构与前端 api.md 对齐
   • 输出：接口设计确认单（含字段映射、校验规则、错误码）

         ↓
③ service-codegen（服务层代码生成）
   输出：
   • Controller（路由 + 参数校验 + 权限注解）
   • Service 接口 + ServiceImpl（业务逻辑骨架）
   • Mapper / Repository（数据访问层）
   • Entity / DTO / VO（数据模型）

         ↓
④ db-migration（数据库变更，有 Schema 变更时触发）
   输出：
   • DDL 变更脚本（CREATE / ALTER）
   • 回滚脚本（DROP / 反向 ALTER）
   • 变更说明文档

         ↓
⑤ unit-test-gen（单元 / 集成测试生成）
   • 基于 api.md 契约生成接口测试用例
   • 生成 Service 层 Mock 单测
   • 生成边界值和异常路径用例
   输出：src/test/ 测试套件

         ↓
⑥ convention-audit-be（后端规范审计）
   扫描：
   • 分层规范（Controller 不含业务逻辑 / Service 不含 SQL）
   • 命名规范（驼峰 / 模块前缀 / 接口命名）
   • 异常处理规范（统一 Result 包装 / 全局异常处理）
   • 事务注解规范（@Transactional 粒度）
   • 日志规范（关键操作必须有操作日志）
   输出：结构化后端审计报告

         ↓
⑦ code-fix-be（可选）
   → convention-audit-be 复扫确认

         ↓
⑧ 输出：可部署服务 + 测试套件 + 审计报告 + 变更脚本
```

### 5.2 后端 Agent 角色设计

```
┌──────────────────────────────────────────────────────────────┐
│                  Orchestrator-BE（后端调度层）                   │
│  与前端 Orchestrator 共享上层调度，或作为独立子 Orchestrator        │
└───────────┬──────────────────────────────────────────────────┘
            │
  ┌─────────▼──────────┐      ┌────────────────────┐
  │  接口设计 Agent      │      │   代码生成 Agent     │
  │  api-design-be     │─────►│   service-codegen   │
  │  → 接口确认单        │      │   → Controller      │
  │    + 字段映射表      │      │     Service / DAO   │
  └────────────────────┘      │     Entity / DTO    │
                               └──────────┬─────────┘
                                          │
  ┌─────────────────────┐     ┌──────────▼─────────┐
  │   审计 Agent         │◄────│   测试生成 Agent     │
  │   convention-audit  │     │   unit-test-gen     │
  │   code-fix-be       │     │   → 测试套件          │
  │   → 审计报告          │     └────────────────────┘
  └─────────────────────┘
            │
  ┌─────────▼──────────┐
  │   数据库变更 Agent   │  ← 此节点必须有人工确认后才执行
  │   db-migration      │
  │   → DDL 脚本        │
  │     + 回滚脚本       │
  └────────────────────┘
```

### 5.3 场景分支

| 场景 | 走哪段链路 |
|---|---|
| 新模块全量开发 | 全链路 ① → ⑧ |
| 已有接口，只新增字段 | ③（只改 DTO/VO）→ ④（加字段 DDL）→ ⑤ → ⑥ |
| 存量服务体检 | 直接 ⑥，输出审计报告，再按优先级修复 |
| 只做接口对齐（前后端联调） | ② 确认对齐 → ③ 局部调整 |

### 5.4 与前端工作流的衔接点

| 衔接点 | 说明 |
|---|---|
| `api.md` | 前端产出，后端消费；是前后端 Agent 协作的核心契约文件 |
| `business-doc` | 前端 `business-doc-extract` 产出的业务理解文档，后端 `api-design-be` 可复用 |
| 审计报告 | 前后端各自产出审计报告，测试 Agent 统一消费，生成回归测试点 |
| 菜单/权限同步 | 后端代码 + 前端页面都就绪后，统一触发 `menu-sync / permission-sync` |

### 5.5 质量红线

- 所有 Controller 接口必须有对应的 api.md 契约，无契约不合并主干
- DDL 变更脚本必须附带回滚脚本，不允许裸跑无回滚的变更
- Service 层不允许直接写 SQL，必须通过 Mapper/Repository
- 写数据库变更必须有人工确认，不允许 Agent 自动执行 DDL

## 七、里程碑速览

| 阶段 | 目标 | 完成标志 |
|---|---|---|
| 🟡 当前 · L5 前端验证 | 前端 Pipeline 跑通跑稳 | 3+ 页面全链路无返工 |
| 🔭 近期 · L5 后端启动 | 后端工作流落地（接口→代码→测试） | 1 个完整后端模块走通链路 |
| 🔭 近期 · L6 最小原型 | 前端 2 个 Agent + Orchestrator | 自动串联，写操作有确认机制 |
| 🔭 中期 · L6 完整 + 工作台 | 前后端完整角色 Agent + 可视化工作台 | 节点状态可视、人工确认入口可用 |
| 🔭 远期 · 全链路闭环 | 产品 + 测试 Agent 接入 | 从需求到测试报告一键触发 |
| 🔭 未来 · L7 | 自演化：审计报告 → Skill 自优化 | 体系能根据历史数据改进规则 |

## 八、一句话总结

> 开发侧先行打通，后端同步建设，可视化工作台保障可观测性；  
> 链路稳定后，产品和测试自然接入——这是从单点提效到全流程自动化最务实的路径。
