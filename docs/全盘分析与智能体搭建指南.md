| 项目感知 | `wls_route_check` | 检查页面目录是否在路由文件中可发现 |
| 项目感知 | `wls_git_log_extract` | 提取近期 Git 提交摘要 |
| 通知 | `wls_audit_report_push` | 推送最新审计报告到飞书 webhook（可选） |

### 3.1 MCP 扩展价值

- **`wls_code_scan`**：解决 AI 只能猜目录结构的问题，是 convention-audit 和 Pipeline 的关键前置能力。
- **`wls_route_check`**：让 page-codegen/menu-sync 后有闭环验证，不再只看文件是否生成。
- **`wls_git_log_extract`**：支撑 Git 规范审计与后续 changelog-gen。
- **`wls_audit_report_push`**：把审计结果从本地报告推进到团队协作渠道。

## 4. CLI 现状

| 命令 | 用途 | 适用场景 |
|---|---|---|
| `init` | 全量安装 AI 体系 | 新项目接入 |
| `update` | 增量更新 kit 文件 | 已接入项目升级 |
| `clean` | 清理开发期 AI 文件 | 构建前/交付前 |
| `check` | 环境预检 | 新成员接入、问题排查 |
| `diff` | 对比已安装文件与当前 kit | update 前评估影响 |
| `validate` | 静态扫描页面文件完整性 | CI 或审计前快速检查 |
| `export` | 导出菜单/字典/权限基线 xlsx | 给产品/后端/测试查看 |

### 4.1 CLI 扩展价值

- `check` 降低新成员配置成本。
- `diff` 降低团队升级时的不确定性。
- `validate` 提供无 AI 的最低限度质量门禁。
- `export` 让 reports 数据更容易被非研发角色使用。

## 5. Agent Pipeline 协议

已新增：

```text
files/.github/skills/_pipeline.md
```

该文件定义：

- 每个 Skill 的 `input_from`
- 每个 Skill 的 `output_file`
- 完成后的 `next_suggest`
- Agent 完成摘要格式
- 哪些动作必须等待人工确认

推荐主链路：

```text
prototype-scan
  → api-contract
  → page-codegen
  → convention-audit
  → code-fix（可选）
  → menu-sync / dict-sync / permission-sync（按需）
```

## 6. 如何搭建智能体

### Step 1：安装与环境预检

```bash
npx @robot-admin/git-standards init
npx @agile-team/wl-skills-kit@latest
npx @agile-team/wl-skills-kit@latest check
```

检查点：

- Node 版本 >= 16
- `.prettierrc.js` / `eslint.config.ts` / `.husky/` 存在
- `.github/skills/sync/env.local.json` 已填写（如需后端同步）
- MCP server 能被编辑器识别

### Step 2：让 Agent 读取调度入口

Agent 每次执行任务前应优先读取：

```text
.github/copilot-instructions.md
.github/skills/_registry.md
.github/skills/_pipeline.md
.github/standards/index.md
```

目的：

- `_registry.md` 决定触发哪个 Skill
- `_pipeline.md` 决定下一步建议
- `standards/index.md` 决定按任务懒加载哪些规范

### Step 3：先做项目感知

建议在复杂任务开始前调用：

```text
wls_code_scan({ path: "src/views" })
wls_git_log_extract({ n: 20 })
```

产出：

- 页面目录清单
- 文件完整性统计
- `API_CONFIG` 分布
- 近期提交摘要

### Step 4：按 Pipeline 执行

有原型/详设：

```text
prototype-scan → api-contract → page-codegen → convention-audit
```

只有口述需求：

```text
page-codegen → convention-audit
```

接手存量项目：

```text
wls_code_scan → convention-audit → code-fix → convention-audit 复扫
```

涉及菜单/字典/权限：

```text
menu-sync / dict-sync / permission-sync
```

### Step 5：每步完成必须输出下一步建议

Agent 完成任一 Skill 后，必须输出：

```text
## 完成摘要
- 产物：...
- 关键数据：...
- 风险：...

## 建议下一步
- next_suggest：...
- 原因：...
- 是否需要用户确认：是
```

### Step 6：闭环验证

建议组合：

```bash
npx @agile-team/wl-skills-kit@latest validate
```

MCP 验证：

```text
wls_route_check({ path: "src/views" })
wls_audit_report_push({})  // 可选，需 feishu_webhook
```

最终验证：

- 审计报告无阻断项
- 关键页面能在路由中发现
- 菜单/字典/权限同步摘要无失败项
- Git 提交符合团队规范

## 7. 下一阶段建议

### 7.1 短期（当前可做）

- 在真实业务项目中试跑 `wls_code_scan → convention-audit → code-fix → 复扫`
- 在 1~2 个页面上验证 `prototype-scan → api-contract → page-codegen → convention-audit`
- 将 `wl-skills validate` 接入 CI 的非阻断检查阶段

### 7.2 中期（L5 稳定后）

- 增加 `changelog-gen` Skill，复用 `wls_git_log_extract`
- 增加 `api-impact-scan` Skill，扫描 api.md 变更影响面
- 增加 `prototype-diff` Skill，辅助原型迭代比对

### 7.3 暂不建议

- 暂不做 Multi-Agent 编排框架
- 暂不做 sale/produce 领域专属 Skill
- 暂不做规范自演化自动合并，只保留人工 review

---

## 8. 结论

当前 wl-skills-kit 已经从“规范文档包”升级为“Skills + MCP + CLI + Pipeline 协议”的智能体基础设施。

下一步不是盲目增加领域模板，而是先把通用 Agent Pipeline 跑稳：

```text
项目感知 → Skill 调度 → 自动生成/审计/修复 → 同步后端 → 复扫闭环
```

当这条链路在多个项目中稳定后，再考虑沉淀领域 Skill 或 Multi-Agent。
